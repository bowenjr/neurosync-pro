# Raspberry Pi 3 deviation notes

The verified deployment target for this project is a **Raspberry Pi 3 Model
B Rev 1.2**, hostname `neurosync-pi`, static address `10.0.0.127`, running
`aarch64` Raspberry Pi OS Lite 64-bit. Rev 5.4 nominally assumes a
Raspberry Pi 5, but this repository intentionally uses the Pi 3 target as
an owner-approved hardware deviation because the Raspberry Pi 5 is
dedicated to another project.

> Note on OS naming: the machine-probed manifest
> (`hardware/manifests/raspberry-pi.json`) records the OS as
> `Debian GNU/Linux 13 (trixie)` — this is the same system. Current
> Raspberry Pi OS (64-bit) is Debian 13 "trixie"-based, and the probe reads
> the OS-release `PRETTY_NAME`, which reports the Debian base rather than
> the "Raspberry Pi OS" marketing name. The two strings describe one OS;
> the manifest value is the authoritative probed one.

This document exists so that assumptions don't quietly drift toward Pi 5
capabilities while the actual production candidate remains the verified
Pi 3.

## Acceptance gate

The Pi 3 target must pass a performance gate before the production
Qt/Cage HMI is accepted: real-time audio must remain stable under the
Phase 8/9 workload. Until that gate passes, the Pi 3 remains the intended
target, but production HMI acceptance is blocked.

## Why this matters

- **CPU**: Pi 3 is a quad-core Cortex-A53 @ ~1.2 GHz (BCM2837), materially
  slower than a Pi 5. Avoid workloads on the Pi that assume Pi 5-class
  headroom (e.g., heavy in-process signal processing) — prefer pushing
  timing-critical or compute-heavy work to the ESP32 or back to Armoury for
  offline analysis.
- **RAM**: Pi 3 tops out at 1 GB. Do not assume multi-GB working sets are
  available on the Pi; logging and buffering code should be written for a
  constrained-memory target and should flush/rotate rather than accumulate.
- **USB**: Pi 3 USB is shared with onboard Ethernet/USB hub silicon (single
  upstream USB channel on early revisions), so USB serial throughput and
  latency budgets should stay conservative.
- **No onboard Bluetooth/Wi-Fi assumptions beyond Pi 3's actual chip**: Pi 3
  has BCM43438 Wi-Fi/BT; don't assume Pi 5's newer radio or PCIe-attached
  peripherals.
- **No PCIe, no native power button, no Pi 5-only GPIO features.**
- **Python version**: this repo pins `requires-python = ">=3.11"` precisely
  because the Pi's OS-provided Python may lag behind whatever is newest on
  Armoury. Do not raise the floor without first confirming what the actual
  deployed Pi provides (`neurosync pi-info`, `scripts/pi/inventory.sh`).

## Practical implications for this repo

- Keep the Pi-side supervisory app's dependency footprint light. Heavy
  numerical/GUI dependencies are deliberately deferred (see Phase 4 of the
  setup) until there's a concrete need, and even then should be evaluated
  for Pi 3 viability before being added to the shared `pyproject.toml`.
- `scripts/pi/bootstrap.sh` detects and records the exact Pi model,
  OS, architecture, Python, storage, and memory into
  `hardware/manifests/raspberry-pi.json` — treat that file as the source of
  truth for what the real target actually is, rather than assuming Pi 3
  nominal specs.

## Capability assessment (why the Pi 3 is sufficient, not a compromise)

The Pi 3 is a validated target for this prototype, not a fallback. The
reasoning, by subsystem:

- **40 Hz timing and cross-feature synchronization do not depend on the
  Pi at all.** All deterministic timing — the 40 Hz waveform, the sync
  pulse, optical PWM phase — is generated by the ESP32's hardware
  timers/MCPWM. The USB serial link is control-plane only; a serial packet
  never clocks a cycle. A Pi 3 and a Pi 5 produce identical timing
  precision here because neither is in the timing loop. Switching to a Pi 5
  would buy zero timing improvement.
- **Real-time audio rides I2S, bypassing the Pi 3's known weakness.** The
  PiFi DAC+ (PCM5122) is an I2S device on the GPIO header, not a USB-audio
  dongle, so it avoids the shared USB/Ethernet hub that is the Pi 3's usual
  audio bottleneck. Generating binaural/monaural/isochronic tones is
  lightweight DSP well within the Pi 3's envelope. Stability is confirmed
  by the Phase 8/9 gate rather than assumed, but the gate is expected to
  pass.
- **The HMI is a lightweight pre-run selection form** (choose function(s)
  to run, duration, audio-clip/beat selection, start/stop) — a static
  control panel, not a live-updating, redraw-heavy GUI. This is
  comfortably within the Pi 3's capability whether implemented as native
  Qt or a served web UI.

Conclusion: no requirement of this prototype — timing, synchronization,
audio, or HMI — necessitates a Pi 5. The Pi 5 is intentionally reserved
for another project. The Phase 8/9 audio-stability gate stands as
confirmation of the audio path, not as a gate expected to fail.
